## What this release delivers

v7.0.0 closes the four competitive gaps identified in the May 26 competitive evaluation and brings ChurchCore Care to feature parity with the market's best general-purpose EHR platforms on every dimension except native iOS/Android apps. All code for AI session notes, EDI claim submission, insurance eligibility verification, mobile access, group therapy, analytics, and billing reconciliation is now shipped and tested.

## Phase C ‚Äî Mobile PWA

**C1 ‚Äî PWA scaffold** (`apps/mobile/`)

- Vite + React 19 PWA with Web App Manifest, service worker, and offline caching via Workbox
- Bottom navigation shell (Schedule, Search, Notes, More)
- Install prompt on iOS/Android (Add to Home Screen)
- `pwa-counselor` RBAC role; counselor-scoped session auth

**C2 ‚Äî Mobile auth**

- Login page and session restoration for mobile shell
- Idle timeout and secure session handling mirroring web app

**C3 ‚Äî Today's schedule**

- `TodaySchedulePage` ‚Äî counselor's daily appointment list
- Tap to expand: client name, time, type, join telehealth link
- Real-time fetch from `/api/v1/appointments`

**C4 ‚Äî Client quick search**

- `ClientSearchPage` ‚Äî live search with debounce, PHI-safe (name only)
- Tap to open client summary card with contact and last-session info

**C5 ‚Äî Quick note entry**

- `QuickNotePage` ‚Äî offline-capable note drafting with IndexedDB queue
- Syncs to `/api/v1/progress-notes` when connectivity restores
- SOAP/DAP/BIRP format selector

**C6 ‚Äî Push notifications**

- VAPID-based Web Push via `apps/worker/src/push-sender.js`
- `push_subscriptions` table; worker polls and dispatches appointment reminders
- Opt-in permission flow in mobile shell
- `nav.groups` i18n key; VAPID key pair required (admin task)

## Phase D ‚Äî Analytics & Reporting

**D1 ‚Äî Analytics API** (`apps/api/src/lib/analytics.js`)

- `getSessionVolume` ‚Äî daily buckets, counselor filter, preset ranges (week/month/quarter/year)
- `getRevenueStats` ‚Äî billed / collected / outstanding from claims
- `getNoShowRate` ‚Äî total + no-show count + rate %
- `getOutcomeTrends` ‚Äî PHQ-9 (or any scored form) time-series
- `getCounselorProductivity` ‚Äî per-counselor sessions / completions / no-shows / avg duration
- CSV builders: `buildSessionsCsv`, `buildRevenueCsv`
- All functions enforce `tenantId`; return empty/zero in in-memory mode

**D2‚ÄìD5 ‚Äî Analytics dashboard** (`apps/web/src/components/Analytics/`)

- `AnalyticsDashboard` ‚Äî preset selector, 4 stat cards, charts, CSV export buttons
- `SessionVolumeChart` ‚Äî AreaChart (`@mantine/charts`) for daily volume
- `OutcomeTrendChart` ‚Äî LineChart with PHQ-9 severity bands (referenceLines at 5/10/15/20)
- `CounselorProductivityTable` ‚Äî sortable columns, click-to-toggle asc/desc
- `StatCard` ‚Äî metric card with trend arrow (TrendingUp / TrendingDown / Minus)
- `AgingReport` ‚Äî six-bucket AR aging display
- Analytics nav item added for admin role

**Tests:** `analytics.test.mjs` ‚Äî 14 tests (in-memory states, CSV builders, date-range parsing)

## Phase E ‚Äî Billing Reconciliation

**E1 ‚Äî ERA parsing + worker reconciler**

- `apps/api/src/lib/era-parser.js` ‚Äî `parseEra(payload)`: Stedi 835 JSON parser; derives `eraStatus` (paid / partially_paid / denied) and adjustment reason (`CO-45` format)
- `apps/worker/src/era-reconciler.js` ‚Äî polls Stedi for pending ERAs, matches by claim ID, posts paid amount + adjustment reason; idempotent via `era_received_at`; acknowledges ERA only after DB write succeeds
- DB migration: `claims.paid_amount NUMERIC(10,2)`, `claims.adjustment_reason VARCHAR(256)`

**E2 ‚Äî Payment posting UI**

- `EraReconciliationBadge` ‚Äî teal/yellow/red badge with ERA date and adjustment code
- `ClaimCard` ‚Äî shows paid amount, adjustment reason, ERA date
- `ClaimsList` ‚Äî status filter dropdown (all / draft / submitted / accepted / paid / partially_paid / denied / rejected)

**E3 ‚Äî Aging report**

- `AgingReport` ‚Äî six-bucket aging (Current, 1‚Äì30, 31‚Äì60, 61‚Äì90, 90+, Total); added to AnalyticsDashboard

**E4 ‚Äî Patient statements**

- `apps/api/src/lib/statement-generator.js` ‚Äî `generateStatement()`: print-ready HTML; practice header, claims table, balance totals; PHI decrypted only for this response, never logged
- `GET /v1/clients/:id/statement` ‚Äî streams HTML; billing role required; audit event on every call
- `BillingTab` added to `ClientDetailTabs`

**Tests:** `era-parser.test.mjs` (6 tests), `statement-generator.test.mjs` (9 tests)

## Phase F ‚Äî Group & Family Therapy

**F1 ‚Äî Data model**

- DB migration: 7 new tables ‚Äî `therapy_groups`, `group_members`, `group_sessions`, `group_session_notes`, `group_member_notes`, `relational_units`, `relational_unit_members`
- All tables tenant-scoped; note content encrypted at rest
- `apps/api/src/db/queries/groups.js` ‚Äî full CRUD with `DB_MODE()` in-memory guards

**F2 ‚Äî Group scheduling**

- `POST /v1/groups`, `GET/PATCH /v1/groups/:id` ‚Äî group lifecycle
- `POST /v1/groups/:id/sessions` ‚Äî schedule with RRULE support
- `expandRrule(dtstart, rruleStr, maxCount=52)` ‚Äî native RRULE (no npm dependency); FREQ=WEEKLY with INTERVAL, COUNT, BYDAY, UNTIL; caps at 52 sessions

**F3 ‚Äî Group session notes**

- `GET/POST /v1/groups/sessions/:id/notes` ‚Äî shared note + per-member notes
- `GroupSessionNotesForm` ‚Äî shared textarea + collapsible per-member sections; SOAP/DAP/BIRP format selector
- `GroupDetailPage` ‚Äî sessions tab (schedule, cancel, notes, bill) + members tab (add/remove)
- `GroupsPage` ‚Äî list with create modal; wired into counselor and admin nav

**F4 ‚Äî Relational units**

- `GET/POST /v1/relational-units` ‚Äî couples/family/other unit lifecycle
- `GET /v1/clients/:id/relational-units` ‚Äî units visible in clinical chart
- `RelationalUnitCard` ‚Äî unit type badge + label in clinical chart (returns null when empty)

**F5 ‚Äî Group billing**

- `apps/api/src/lib/group-billing.js` ‚Äî CPT-coded claim generation per active member
  - 90853 (group therapy), 90849 (multiple family), 90847 (conjoint/couple)
- `POST /v1/groups/sessions/:id/billing/submit` ‚Äî idempotent via `notes` column marker; returns `{ claimIds, alreadySubmitted }`

**Tests:** `groups.test.mjs` (12 tests ‚Äî in-memory stubs + expandRrule), `group-billing.test.mjs` (6 tests)
