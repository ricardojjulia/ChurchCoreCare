**Status:** ⚠️ UNTESTED — Under Review
**Date:** 2026-03-31

---

## What's New

### 1. High-Touchpoint Inline Toggle (Client Workspace)

The high-touchpoint flag is no longer a read-only badge. Each client row now shows a clickable toggle button — filled grape when active, outlined when inactive. One click PATCHes the server immediately with an optimistic UI update and rollback on failure.

### 2. Sessions Button per Client (Client Workspace)

A new "Sessions" button appears on every client row (between Edit and Schedule). Clicking it opens a modal with the client's full appointment history, newest first, showing: Date & Time · Type · Status · Counselor · Duration.

### 3. Client Sessions Tab (Scheduling Workspace)

A new "Client Sessions" tab in the Scheduling workspace lets any user:

1. Search and select a client
2. View their full appointment history in a table
3. See note status on each past appointment:
   - **Notes Filed** (teal) — at least one locked progress note linked to that appointment
   - **Draft — Pending Sign-off** (yellow) — draft note exists, not yet locked
   - **No Notes** (red) — no progress note linked
   - **Upcoming** (blue) — future appointment, no note status shown
4. Click **View Chart** on any appointment that has notes to navigate directly to the Clinical Chart with that client pre-selected

### 4. Clinical Chart Deep-Link (`initialClientId` Prop)

`ClinicalChartPage` now accepts an `initialClientId` prop. `App.jsx` manages a `clinicalChartInitialClientId` state value (cleared when navigating away from Clinical Chart) and passes it through. The `handleOpenClinicalChart(clientId)` handler in App.jsx wires Scheduling → Clinical Chart navigation.

## No API Changes

All endpoints were already available from v5.3.x:

- `PATCH /v1/clients/:id` — accepts `highTouchpoint` boolean
- `GET /v1/appointments?clientId=:id` — client appointment history (added v5.3.2)
- `GET /v1/clients/:id/progress-notes` — note status per appointment
